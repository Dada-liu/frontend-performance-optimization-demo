## 存在的问题

1. 首页加载了大量未压缩的图片（多为PNG格式，单张图片体积超过2MB），且未使用懒加载，首屏加载时一次性请求所有图片资源；

2. JS和CSS资源未压缩、未拆分，主JS文件体积达1.2MB，且未配置CDN，资源加载延迟严重；

3. 首屏关键CSS未内联，外部CSS文件加载延迟导致首屏渲染阻塞；

4. 存在未使用的第三方依赖（如完整引入的lodash库），冗余代码过多。


## 优化后分析

1、图片加了 懒加载 后，并没有对首页加载速度有提升，问题出在即使只有视口内的几张图片，它们还是很大，加载需要很久的时间；

2、图片压缩：图片改用 jpg 后，速度明显提升

3、开启资源缓存：LCP 从 0.2 能达到 0.1
现代前端项目最优方案：
    1. 文件名带哈希的静态资源（app.8f2a.js） + 强制开启永久缓存（Cache-Control）
    2. 入口文件（index.html），禁用强缓存，只保留协商缓存，保证及时更新

4、（压缩只针对字节文件，不针对二进制文件）开启 gzip/brotli 压缩：

| 压缩方式 | index.html（kb） | main.js（kb） | styles.css（kb） |
| -------- | ---------------- | ------------- | ---------------- |
| 无压缩   | 0.8              | 11.2            | 2.0              |
| gzip     | 0.7               | 3.5            | 0.9               |
| brotli   | 0.6               | 3.2            | 0.8               |


5、使用 esbuild 压缩代码

| 压缩方式         | main.js（kb） | styles.css（kb） |
| ---------------- | ------------- | ---------------- |
| 无压缩           | 11.2          | 2.0              |
| esbuild          | 6.2           | 1.6              |
| esbuild + gzip   | 2.4           | 0.9              |
| esbuild + brotli | 2.3           | 0.8              |

6、将首屏关键CSS内联

## 相关知识点

1、 浏览器原生页面生命周期函数的执行顺序
1. DOMContentLoaded：HTML解析完成、DOM 树构建完毕
2. load：页面所有资源（图片、视频、字体、css、js）全部加载完成
3. beforeunload：页面即将卸载/刷新/关闭，可以阻止页面关闭
4. unload：页面彻底销毁，资源回收

2、 HTTP 协议版本不一样，对请求并发的限制不一样
1. HTTP/1.1：文本、短连接、6 并发、队头阻塞重灾区
2. HTTP/2：TCP + 二进制 + 多路复用，解决应用层阻塞，解决不了 TCP 丢包阻塞
3. HTTP/3：换 UDP+QUIC，握手更快、无队头阻塞、移动端无敌

在 Nginx 配置中可以修改协议版本；
